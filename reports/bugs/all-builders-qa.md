<div style="border-left:4px solid #b91c1c;background:#fef2f2;border-radius:0 8px 8px 0;padding:16px 20px;margin:12px 0">
<span style="background:#b91c1c;color:#fff;font-size:10px;font-weight:700;padding:2px 8px;border-radius:4px;letter-spacing:.5px">P0</span>&nbsp;<strong>AI chatbox overlay permanently blocks all builder interactions</strong>
</div>

**Severity:** P0 — Critical
**Area:** Functionality / Logic
**Affects:** All 4 builders (12422, 12421, 12416, 12415)

**Issue:** On every widget builder page load, `div.ai-chatbox-overlay` renders in the DOM in an active/intercepting state and captures all pointer events across the entire builder canvas. Every click attempt on any element — CSS tab, Publish button, Add Section, Publish dropdown — fails with `<div class="ai-chatbox-overlay">…</div> intercepts pointer events`. This means the entire builder is functionally unusable: no tab switching, no publishing, no section management, no code editing clicks are possible. This is a P0 because the builder's primary interactive flows are completely blocked.

**Steps to Reproduce:**
1. Log in to wdesignkit.com
2. Navigate to any widget builder URL: `/admin/widgets/widget-builder/12422` (or 12421, 12416, 12415)
3. Wait for the page to fully load (6+ seconds)
4. Attempt to click on the CSS tab, Publish button, or Add Section
5. Observe that clicks do not register on any builder element

**Expected Result:** Builder elements (tabs, buttons, controls) respond to clicks immediately. The AI chatbox overlay should only be visible and interactive when the user explicitly opens it via the "Build with AI" button.

**Actual Result:** `div.ai-chatbox-overlay` permanently occupies the builder viewport and intercepts all pointer events, rendering the entire builder non-interactive.

**Solution:**
- The AI chatbox overlay is initializing in an open/active state rather than a hidden state on page load
- Locate the `ai-chatbox-overlay` component initialization — it likely has a default `isOpen: true` state or missing initial `display: none` / `pointer-events: none`
- Fix: Set the initial state to `isOpen: false` and ensure the overlay applies `pointer-events: none` when `isOpen` is false
- The fix should be applied to the shared builder layout component so it applies across all 4 builder types simultaneously
- Verify with: `getComputedStyle(document.querySelector('.ai-chatbox-overlay')).pointerEvents` should return `"none"` on initial load

<div style="border-left:4px solid #dc2626;background:#fef2f2;border-radius:0 8px 8px 0;padding:16px 20px;margin:12px 0">
<span style="background:#dc2626;color:#fff;font-size:10px;font-weight:700;padding:2px 8px;border-radius:4px;letter-spacing:.5px">P1</span>&nbsp;<strong>Ace worker files return HTTP 404 — files missing from deployment</strong>
</div>

**Severity:** P1 — High
**Area:** Functionality / Console / Performance
**Affects:** All 4 builders — confirmed same 3 failing URLs across every builder

**Issue:** All three Ace Editor language worker files (`worker-css.js`, `worker-javascript.js`, `worker-html.js`) return HTTP **404 Not Found** from the path `/_next/static/chunks/`. The files do not exist at the deployed location. This is worse than a MIME type issue — the files are entirely missing from the build output. The browser logs 3 console errors per builder page load: *"Loading script from worker-*.js with importScripts() was blocked because of a disallowed MIME type (text/plain)"*. Without these workers, Ace Editor runs in degraded mode: no syntax highlighting, no real-time error detection, no code linting — across all 4 builder types.

**Steps to Reproduce:**
1. Open DevTools → Network tab on any builder URL
2. Filter by "worker"
3. Navigate to `/admin/widgets/widget-builder/12422`
4. Observe: `worker-css.js`, `worker-javascript.js`, `worker-html.js` all return 404

**Expected Result:** All 3 Ace worker files return HTTP 200 with `Content-Type: application/javascript`. Ace Editor syntax highlighting and linting are active in the code editor.

**Actual Result:** All 3 worker files return HTTP 404 with `Content-Type: text/plain`. 3 console errors are logged on every builder page load. Syntax highlighting is disabled across all builder types.

**Solution:**
- The Ace worker files are not included in the Next.js build output. They need to be copied to `public/_next/static/chunks/` or configured in `next.config.js` to be bundled
- Check if `ace-builds` package is installed and if its worker files are being referenced with the correct path — Ace worker files often need to be served from a CDN or explicitly copied via a `CopyWebpackPlugin`
- Add to `next.config.js` webpack config: copy `node_modules/ace-builds/src-min-noconflict/worker-*.js` to the public static directory
- Alternatively, use `ace.config.set('basePath', '/path/to/ace-workers/')` to point Ace to where the files are actually deployed

- Locate the Push-type builder header/toolbar template — it appears to use a different component than Core/Bricks

<div style="border-left:4px solid #d97706;background:#fffbeb;border-radius:0 8px 8px 0;padding:16px 20px;margin:12px 0">
<span style="background:#d97706;color:#fff;font-size:10px;font-weight:700;padding:2px 8px;border-radius:4px;letter-spacing:.5px">P2</span>&nbsp;<strong>Panel navigation tabs (Controls, Preview) are DIV/SPAN without semantic role</strong>
</div>

**Severity:** P2 — Medium
**Area:** Accessibility
**Affects:** All 4 builders

**Issue:** The main navigation tabs in the builder left panel — "Controls" (rendered as `<DIV>`) and "Preview" (rendered as `<SPAN>`) — have no `role`, no `tabindex`, and no `aria-selected` attribute. These are primary navigation controls that switch the entire panel content. Without proper tab semantics, screen readers cannot identify them as tabs, cannot navigate between them with Arrow keys, and receive no feedback about which tab is active. This breaks the ARIA tab pattern (WAI-ARIA 1.1 §3.23).

**Steps to Reproduce:**
1. Open any builder and inspect the "Controls" and "Preview" tab elements in the left panel
2. Check for role, tabindex, aria-selected attributes

**Expected Result:** Panel tabs should implement the ARIA tab pattern: parent with `role="tablist"`, each tab with `role="tab"`, `aria-selected="true/false"`, `tabindex="0"` (active) or `tabindex="-1"` (inactive), and ArrowLeft/ArrowRight keyboard navigation.

**Actual Result:** "Controls" is a `<DIV role="NONE" tabindex="NONE">` and "Preview" is a `<SPAN role="NONE" tabindex="NONE">` — no semantic meaning for assistive technologies.

**Solution:**
- Wrap both tabs in `<div role="tablist" aria-label="Builder panel">`
- Add `role="tab"`, `aria-selected="true/false"`, and correct `tabindex` values to each tab
- Add ArrowLeft/ArrowRight key handlers to move focus between tabs and Enter/Space to activate
- Apply in the shared builder panel component to fix all 4 builders at once

<span style="background:#d97706;color:#fff;font-size:10px;font-weight:700;padding:2px 8px;border-radius:4px;letter-spacing:.5px">P2</span>&nbsp;<strong>Push/Save buttons missing aria-label on all 4 builders</strong>
</div>

**Severity:** P2 — Medium
**Area:** Accessibility
**Affects:** All 4 builders

**Issue:** Both the Push button and Save button in the builder toolbar have `aria-label: "MISSING"` across all 4 builder types. While both have visible text labels ("Push" / "Save"), they lack `aria-label` for screen reader context and have no `title` tooltip for sighted keyboard users who Tab to them. More critically, on Push Core and Push Elementor the Push button is a `<SPAN>` (separate P1 bug), making the missing aria-label compound an already broken element.

**Steps to Reproduce:**
1. Open any builder and inspect the Push and Save buttons
2. Check for `aria-label` and `title` attributes

**Expected Result:** Both buttons should have `aria-label` matching their visible label, plus a `title` tooltip. For Push Core: `aria-label="Push Core"`, for Push Elementor: `aria-label="Push Elementor"`.

**Actual Result:** Neither button has `aria-label` or `title` on any of the 4 builders.

**Solution:**
- Add `aria-label="[builder type] push"` to the Push button and `aria-label="Save widget"` to the Save button
- Add `title` attribute with the same value for tooltip display
- Apply in the shared toolbar component and also in the Push-type builder templates

**Severity:** P2 — Medium

**Affects:** All 4 builders

**Issue:** None of the 4 builder pages contain a back button or navigation link to return to the Widgets list (`/admin/widgets/uploaded`).
